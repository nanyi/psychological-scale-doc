## 2. 概述

### 2.1 模块定位

量表订单与支付模块是心理测评系统的核心商业化模块，负责量表的购买、支付、退款等核心交易流程，支持个人购买和企业团购两种商业模式，集成微信支付和支付宝支付渠道。

### 2.2 功能摘要

本模块实现量表商品的配置与定价、用户购买下单、订单支付、订单退款等全链路交易功能，支持一个订单包含多个量表，支持对订单中的单个或多个量表进行部分退款。

## 3. 业务需求

### 3.1 功能需求

#### FR-001: 量表单价配置

**功能描述**: 管理员可为每个量表设置销售价格，支持原价、现价、折扣价设置，支持限时折扣活动配置

**输入**:
- 量表ID
- 价格信息（原价、现价、折扣开始时间、折扣结束时间、折扣比例）

**处理逻辑**:
1. 管理员在量表管理后台设置量表价格
2. 系统根据当前时间判断是否在折扣期，返回实际销售价格
3. 价格设置成功后更新量表商品信息

**输出**: 量表价格配置结果

**业务规则**:
- 原价必须大于0
- 现价不能大于原价
- 折扣时间必须符合逻辑（开始时间早于结束时间）
- 未设置价格时量表不可购买

#### FR-002: 购物车管理

**功能描述**: 用户可将多个量表加入购物车，在购物车中统一结算

**输入**:
- 用户ID
- 量表ID列表

**处理逻辑**:
1. 用户选择量表加入购物车
2. 系统校验量表是否可购买（下架、已删除的量表不可加入）
3. 购物车中显示每个量表的名称、价格、数量
4. 支持从购物车移除量表

**输出**: 购物车列表、总价

**业务规则**:
- 购物车最多添加20个量表
- 同一量表只能添加1次
- 未登录用户购物车数据暂存30天

#### FR-003: 订单创建

**功能描述**: 用户从购物车结算或直接购买量表，生成量表订单

**输入**:
- 用户ID
- 量表ID列表（从购物车或直接购买）
- 支付方式（微信支付/支付宝）
- 订单类型（个人购买/企业团购）

**处理逻辑**:
1. 系统根据选择的量表计算订单金额
2. 生成唯一订单编号（格式：ORD+时间戳+随机6位）
3. 创建订单主记录和订单项记录（每个量表一个订单项）
4. 订单状态设置为"待支付"
5. 订单有效期设置（默认24小时）

**输出**: 订单信息、支付参数

**业务规则**:
- 订单编号全局唯一
- 订单金额 = Σ（量表单价 × 数量）
- 订单创建时锁定库存（量表配额）
- 超时未支付自动取消订单

#### FR-004: 订单支付

**功能描述**: 用户完成订单支付，支持微信支付和支付宝支付

**输入**:
- 订单编号
- 支付渠道
- 支付结果回调信息

**处理逻辑**:
1. 统一下单：向支付渠道发起支付请求，获取支付参数（二维码链接/预支付ID）
2. 用户扫码/确认支付
3. 支付渠道异步回调支付结果
4. 系统接收回调，校验签名，更新订单状态
5. 解锁库存，发放量表使用权

**输出**: 支付参数、支付结果

**业务规则**:
- 支付签名校验失败拒绝更新订单状态
- 回调必须幂等处理（重复回调只处理第一次）
- 支付成功后自动发放量表权限

**微信支付请求示例**:
```
POST /api/payment/wechat/pay
{
  "orderNo": "ORD202603091234567890",
  "amount": 99.00,
  "description": "购买SCL-90量表",
  "openid": "oxxxxxx"
}
```

**微信支付响应示例**:
```
{
  "code": 200,
  "data": {
    "orderNo": "ORD202603091234567890",
    "prepayId": "wx12345678901234567890",
    "qrCode": "weixin://wxpay/bizpayurl?pr=xxxxx",
    "expireTime": 7200
  }
}
```

#### FR-005: 订单取消

**功能描述**: 用户主动取消订单或系统超时自动取消订单

**输入**:
- 订单编号
- 取消原因

**处理逻辑**:
1. 校验订单状态（只能取消待支付订单）
2. 更新订单状态为"已取消"
3. 释放锁定的量表配额
4. 如已使用优惠券，释放优惠券使用状态

**输出**: 取消结果

**业务规则**:
- 已支付订单不可取消
- 已退款订单不可取消
- 取消后原订单编号不可复用

#### FR-006: 退款申请

**功能描述**: 用户可对已购买的量表申请退款，支持部分退款（订单中单个或多个量表）

**输入**:
- 订单编号
- 退款订单项列表（订单项ID列表）
- 退款原因

**处理逻辑**:
1. 校验订单状态（已支付的订单才能退款）
2. 校验退款订单项（只能选择未退款的订单项）
3. 计算可退款金额（原价 - 已退款金额）
4. 创建退款记录
5. 调用支付渠道退款接口
6. 更新订单项退款状态
7. 更新订单主状态（部分退款/全部退款）
8. 回收量表使用权

**输出**: 退款结果

**业务规则**:
- 已完成测评的量表不可退款（需人工审核）
- 退款申请需在购买后7天内提出
- 退款金额原路返回
- 退款手续费由用户承担（支付渠道收取的部分）

**退款记录数据结构**:

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGINT | 退款ID |
| refund_no | VARCHAR(32) | 退款编号，唯一 |
| order_no | VARCHAR(32) | 原订单编号 |
| order_item_id | BIGINT | 订单项ID |
| refund_amount | DECIMAL(10,2) | 退款金额 |
| refund_reason | VARCHAR(500) | 退款原因 |
| refund_status | TINYINT | 退款状态：0-处理中，1-退款成功，2-退款失败 |
| refund_channel | TINYINT | 退款渠道：1-微信，2-支付宝 |
| refund_time | DATETIME | 退款时间 |
| create_time | DATETIME | 申请时间 |

#### FR-007: 企业团购

**功能描述**: 企业客户批量购买量表配额，分配给员工或学生使用

**输入**:
- 企业ID
- 量表ID列表及数量
- 分配用户列表

**处理逻辑**:
1. 企业管理员创建团购订单
2. 企业支付订单
3. 系统生成量表配额（企业账户余额）
4. 企业管理员分配量表给指定用户
5. 被分配用户获得量表使用权

**输出**: 团购订单、配额分配结果

**业务规则**:
- 企业配额有效期1年
- 配额不可转让、不可提现
- 员工离职后回收未使用配额

## 4. 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 订单创建 | ≤500ms | 从选择量表到生成订单 |
| 支付跳转 | ≤300ms | 统一下单响应时间 |
| 回调处理 | ≤200ms | 支付回调处理时间 |
| 退款处理 | ≤1s | 退款申请到结果返回 |
| 并发支付 | ≥100TPS | 支持100个并发支付 |
| 订单查询 | ≤200ms | 订单列表和详情查询 |

## 5. 可用性需求

- **SLA要求**: 支付接口可用性≥99.9%，月度累计不可用时间≤43分钟
- **降级策略**: 支付渠道故障时自动切换到备用渠道，紧急情况下支持人工处理
- **超时处理**: 订单支付超时自动取消，支付中状态订单定时校对

## 6. 安全性需求

- 支付签名验证：所有支付回调必须验证签名，防止伪造
- 金额校验：订单金额必须服务端计算，禁止客户端传入
- 幂等处理：支付、退款等操作必须支持幂等
- 日志审计：所有支付、退款操作记录详细日志
- 数据加密：订单敏感信息（金额等）加密存储
